# 🎮 游戏智能自动化测试 Agent 技术方案

## 一、 整体架构概述

本系统采用 **ReAct (Reasoning and Acting)** 架构为核心驱动，将传统写死的线性测试脚本升级为“状态驱动”的智能体。系统分为四大核心引擎：
1. **冷启动构图引擎 (Cold-Start Graph Engine):** 用于前期游戏 UI 状态机的自主探索与缓存。
2. **多模态感知引擎 (Perception Engine):** 负责高效提取屏幕截图、Poco UI 树及运行时状态。
3. **ReAct 决策大脑 (Cognition Engine):** 基于 Gemini 3.1 Pro，负责意图理解、路径规划、异常分析与断言。
4. **工具/技能执行引擎 (Execution Engine):** 封装 Airtest 底层 API，提供原子操作与宏观技能。

---

## 二、 核心模块详细设计

### 1. 认知模块：基于 LangChain 的 ReAct 循环
大模型不直接输出死板的代码，而是通过 `Thought -> Action -> Observation` 的闭环来推进任务。

* **Thought (思考与状态诊断):** Gemini 解析当前传入的 UI 数据，判断当前处于图结构（状态机）的哪个节点，并思考下一步策略。
* **Action (调用工具箱):** 模型通过 Function Calling 输出结构化指令。工具箱分为两层：
    * **原子动作 (Micro-tools):** `poco_click`, `airtest_touch_pos`, `swipe`, `wait_for_node`.
    * **宏技能 (Macro-skills):** `Maps_to(target_node)`（利用图缓存极速跳转）、`clear_all_popups()`（清理弹窗循环）、`upgrade_hero(name, target_level)`。
* **Observation (执行反馈):** Python 端执行工具后，返回执行结果（如：成功到达节点、出现“资源不足”提示），作为下一次 Thought 的上下文。

### 2. 感知模块：成本与延迟优化 (Modality Downgrade)
严格控制 Token 消耗和截图上传频率。

* **L1 级感知 (纯文本/极速):** 默认状态。提取 Poco 树中 `visible=True` 的交互节点，去除冗余布局，转化为 Markdown 列表（如：`- [Button] btn_hero (坐标: 0.1, 0.9)`）。
* **L2 级感知 (多模态/重度):** 仅在以下情况触发截图上传：
    * L1 级文本无法命中任何图节点（遇到未知新界面）。
    * 执行断言审计任务（如扫荡结果 OCR 识别）。
    * 连续 3 次 Action 失败，触发“异常现场保护”。
* **UI Diff 拦截器:** 每次行动后，计算当前截图/Poco 树与上一步的差异度。若界面处于加载动画中，底层引擎执行本地 `sleep`，阻断向 LLM 发送重复请求。

### 3. 冷启动模块：UI 状态机 (MBT) 缓存构建
大幅降低运行时对 LLM 的依赖，实现“系统级”响应速度。

* **图结构定义:** $G = (V, E)$。节点 $V$ 为页面特征哈希（基于 Poco 核心节点集），边 $E$ 为转移该状态所需执行的 Poco 路径或坐标。
* **探索阶段 (DFS 扫图):** 脚本自动化遍历页面，遇到新页面时，调用一次 Gemini 进行语义标注（标注该页面的核心功能和关键按钮）。
* **缓存命中运行 (Cache Hit):** 当 Agent 接到指令（如“去主城收集资源”），系统在本地图数据库中计算最短路径，底层直接连贯执行缓存的 Poco 点击，**全程零 LLM Token 消耗**。

### 三、 核心测试场景落地逻辑 (重构版：全场景泛化智能体)

在这个新架构下，Agent 不再被局限于“爬塔”或“扫荡”，而是转变为一个接收你自然语言指令（如：“创建一个新号，一直推主线到10级，遇到所有抽卡环节都记录下来”）并自动拆解执行的超级玩家。

#### 场景 A：自然语言指令解析与长线主线推进 (The Playthrough)
**目标：** Agent 能够从新建账号开始，理解游戏的新手引导（强制/非强制），并根据你的高层指令自主推进游戏进度。

* **指令拆解 (Task Decomposition)：** 当你输入“一直推主线直到解锁工会系统”，**规划 Agent (Planner)** 会将这个长线任务拆解为子任务栈：`[创建角色 -> 过剧情 -> 完成新手教程 -> 识别任务追踪面板 -> 循环点击主线任务直到目标达成]`。
* **动态寻路与降维执行：** 遇到需要战斗的环节，自动调取底层的 `auto_battle()` 脚本；遇到需要对话的环节，调取 `skip_dialogue()`。
* **未知阻断处理 (自愈合)：** 如果在推主线时弹出了意料之外的“首充提示”或“等级不足请先扫荡升级”，执行流会被打断。此时触发多模态感知，LLM 分析当前画面，插入新的子任务（如：去副本扫荡 5 次升到 10 级），然后再回到原主线任务。

#### 场景 B：伴随式 Bug 探索与现场捕获 (The Critic)
**目标：** 在执行任何“游玩指令”的过程中，Agent 必须具备像专业 QA 一样的“嗅觉”，时刻保持对异常状态的监控。

* **并行审查机制：** 引入一个独立的 **审查 Agent (Critic)**。在执行 Agent 推游戏的同时，Critic 以较低的频率（如每 5 秒或每次场景切换时）抽取画面和 Poco 树。
* **启发式缺陷识别：**
    * **UI 穿模/重叠：** 多模态大模型非常擅长发现 UI 布局混乱或文字溢出框外。
    * **占位符泄漏：** 扫描 Poco 树中是否出现了 `text="null"`, `text="UI_Label_102"`, `Error` 等研发阶段未替换的字符串。
    * **流程死锁：** 如果 Critic 发现游戏画面连续 30 秒没有任何实质性改变（UI Hash 值不变且无动画），判定为卡死 Bug。
* **自动提单：** 一旦发现上述问题，立即抓取日志 (logcat)、前置操作录像/截图、以及复现场景的 Poco 树，自动生成一份格式化的 Bug 报告。

#### 场景 C：指令触发的深度专项测试 (On-Demand Deep Testing)
**目标：** 结合了自动化测试的核心诉求。Agent 在游玩过程中，可以随时停下来执行你下达的特定测试用例。

* **场景流转：** 用户输入：“测试一下背包系统的道具使用逻辑是否正常”。
* **执行逻辑：**
    1.  Agent 通过之前构建的 UI 状态机，导航到“背包”界面。
    2.  读取背包当前数据（如：有 10 瓶经验药水）。
    3.  执行使用操作（点击药水 -> 点击使用）。
    4.  验证背包数据是否变为 9，角色经验值是否相应增加。
    5.  输出专项数值对账报告。

## 四、 技术栈与工程化部署

* **大语言模型层:** Gemini 3.1 Pro API (配置 `response_mime_type="application/json"` 确保 Function Calling 的绝对稳定)。
* **自动化与视觉引擎:** * `poco` (UI 树解析与相对坐标定位)
    * `airtest.core.api` (视觉匹配降级方案与底层物理操控)
    * `OpenCV` (用于前置的轻量级 UI Diff 对比)
* **Agent 编排框架:** `LangChain` 或纯 Python 异步协程 (用于精细控制 ReAct 状态机和上下文滑动窗口)。
* **缓存与知识库:** * SQLite / JSON 文件（用于存储 UI 状态机图结构与冷启动缓存）。
    * 内存级上下文管理器（维护最近 10 步的操作日志，防止 Context Window 爆炸）。

## 五、 实施路线图建议

1. **Phase 1 (基础建设):** 封装 Airtest/Poco 原子 API 为标准化工具；完成前置的过滤与 UI Diff 逻辑。
2. **Phase 2 (冷启动构建):** 编写脚本完成游戏主城、养成、战斗等核心 UI 的哈希图结构缓存。
3. **Phase 3 (Agent 组装):** 接入 Gemini，跑通“任务接收 -> 路径检索 -> 宏指令分发”的极速导航闭环。
4. **Phase 4: AGI 测试管线部署与多智能体协作**

* **阶段 4.1：分层 Multi-Agent 架构搭建**
    * **大脑 (Planner Agent - Gemini 3.1 Pro)：** 利用超大上下文窗口，负责解析你的自然语言指令，维护全局任务栈，并调度底层的原子操作。
    * **眼睛/手 (Actor Agent - Poco/Airtest + 轻量级视觉模型)：** 负责高频、低延迟地执行 Planner 下发的具体指令（如“点击屏幕坐标(0.4, 0.5)”，“查找名为 btn_close 的节点”）。
    * **监工 (Critic Agent - 视觉大模型)：** 专职负责找 Bug、截图存档和生成测试报告。
* **阶段 4.2：引入外部记忆库 (Long-Term Memory / RAG)**
    * 从 0 到 1 玩游戏需要漫长的时间。Agent 必须记住它“昨天”做了什么。
    * 建立一个本地向量数据库或关系型数据库（如 SQLite），记录：【已解锁的功能模块】、【当前主角等级与战力】、【背包核心资产】。当 Planner Agent 规划路线时，必须先查询这个记忆库，避免去点击还未解锁的区域。
* **阶段 4.3：构建自然语言测试 DSL (领域特定语言)**
    * 为了让你更顺畅地下达指令，将 Gemini 的 Function Calling 封装成一套类似自然语言的 API。比如，将底层的 UI 跳转逻辑包装成 `execute_test_case(goal="测试抽卡保底机制", pre_condition="拥有100抽资源")`，让大模型在后台自动将这些高阶指令编译为具体的 Poco 执行动作。
